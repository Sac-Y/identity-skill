# Frontend App Builder

用这个 skill 来创建精致的前端应用、仪表盘、游戏、创意网站、hero 区块、改版设计，以及其他以视觉驱动的 UI。先以资深前端设计师的身份工作，再以工程师身份实现一份已被批准的设计规范（design spec）。

## 核心标准（Core Standard）

本 skill 的两条优先级高于其他一切：

1. 先用 Image Gen 做出足够多、足够好看的设计：干净、通透、有辨识度、完整、可读、必要时按 section 定制，且默认不重复。
2. 在被接受的设计与浏览器实现达到 10/10 匹配之前，不要停下。持续修复视觉、交互、响应式、素材和排版上的偏差，直到 `view_image` 对比能够通过设计公司（agency）的验收标准。

## 硬性规则（Hard Rules）

1. 使用 Image Gen 产出视觉概念，除非用户明确选择退出，或任务只是在既有设计系统内的小型 UI 修复。
2. 在写代码之前先设计完整的目标界面。对于一个完整页面、应用、仪表盘、游戏或产品界面来说，只有 header 或 hero 概念是不够的。对于多 section 网站和长 landing page，优先采用协调一致的逐 section 概念，外加一张可选的总览图来把控节奏，而不是用一张过高的图导致细节丢失。对于应用、仪表盘、游戏或紧凑的产品界面，先生成完整的主屏幕，加上任何需要的状态、响应式或素材概念。
3. 在 Codex 内部，多 section 网站的概念默认为每个主要 section 生成一张全新、大尺寸、可读的 Image Gen 截图。如果需求有 1-10 个 section，就预期大约有 1-10 张主要 section 图。只要文字、按钮、卡片结构、排版、间距或颜色小到无法提取，就额外生成 section/细节截图。不要把旧的整页图裁切或放大来当主参考；重新生成一张全新、独立的 section 或细节图，并保持相同的设计系统。
4. 在 Plan mode 下，先生成设计，然后用 `request_user_input` 取得设计批准，再去规划实现细节。
5. 一旦被接受，该概念就是生产用的设计规范。实现阶段不得自由发挥：不得重新诠释布局、可见文案（copy）、层级、容器模型、样式、图像、密度或 section，除非用户批准，或有具体的阻塞问题要求这么做。通用的设计经验法则永远不能凌驾于被接受的概念之上。
6. 完成门槛是「达到设计公司验收标准的忠实实现」：对被接受规范的 10/10 保真度，加上生产级质量的代码。如果浏览器渲染出的 UI 会收到设计评审意见，就继续修。
7. 写代码之前，从被接受的图里建立一个小型设计系统：tokens、排版、组件族（component families）、变体、间距、图标处理和容器规则。要同时包含内容排版和 UI 框架（chrome）排版，用于工具、编辑器和仪表盘。从这个系统去实现，让重复元素保持一致。
8. 对于全新的复杂应用 UI，例如仪表盘、管理工具、编辑器、数据密集型工具和多面板产品界面，默认使用 React + Vite，除非用户指定了另一个框架、现有仓库已经规定了框架，或任务明确是单文件/静态交付物。
9. 主标题上方的 hero eyebrow、kicker、pretitle、badge 或 pill 标签默认禁止使用。只有在用户明确要求，或被接受/参考的设计里已经包含它时，才使用其中之一。
10. 优先在 Browser 插件 / 内置浏览器里验证。只有在 Browser/IAB 不可用或不可靠时才使用 Playwright Chromium，并说明兜底原因。
11. 在你对被接受的概念和最新的浏览器截图两者都用过 `view_image` 之前，最终交付会被阻塞。这一步不能跳过，也不能仅用浏览器检查来替代。直接评判这一对图：这是否达到了设计公司验收标准的忠实实现，一家优秀、技艺高超的设计公司是否会认可对被接受设计的这一具体实现？如果不会，就继续修。
12. 交付前移除临时的 QA 产物，除非用户或任务明确要求保留它们。

## Image Gen 工作流（Image Gen Workflow）

阅读并遵循已安装的 @imagegen skill。关于网站专用的 briefing 指引，使用 `references/imagegen-website-concepts.md`。

在调用 Image Gen 之前：

- 把用户的具体需求抄进 brief：产品/页面目的、受众、需要的 section 或状态、工作流、提供的文案、导航标签、CTA 标签、数据字段、需要的媒体、响应式需求，以及实现约束。
- 索要完整的目标界面：完整页面、应用屏幕、仪表盘、游戏屏幕，或协调一致的 section/状态集合。如果交付物不止一个 hero，就说明概念必须包含下游 section、状态或响应式延续。如果 section 数量已知或有暗示，就点名每一个需要自己概念截图的 section/状态。
- 重复实现约束：代码原生（code-native）的应用 UI 文字与控件、恰当时带自身文字与品牌的完整渲染产品/背景素材、可分离的素材、可复用的组件族、有意图的容器模型、不用默认卡片网格、不发明 hero eyebrows/kickers/badges/pills、以及务实的 HTML/CSS/组件实现。
- 保留来自用户内容、截图或既有应用的信息架构。不要让 Image Gen 发明不相关的 section、伪造指标、新的产品主张、多余的仪表盘、新的导航或不同的产品叙事。
- 对于多 section 网站或长 landing page，默认每个主要 section 一张协调一致的概念图。可选的总览图只用于结构和节奏；永远不要在一张巨大的压缩拼板会让文字、按钮细节、卡片结构、间距或排版难以分析时依赖它。
- 对于密集的应用、仪表盘、编辑器、产品界面和复杂 section，为那些在单张全屏图里会变得不可读的区域生成单独的状态或细节概念：表格、侧栏、检查器（inspector）、模态框、工具栏、图表、表单、卡片、定价块、用户评价或媒体模块。
- 如果任何概念截图对于实现来说太小、太糊、被裁切、太拥挤或有歧义，就在写代码之前生成一张全新、独立的 section/状态/细节截图。保持相同的配色、排版气质、组件族、素材处理、密度和 section 顺序。不要裁切、切片、放大或复用早先图片的一小部分来当作事实来源。
- 对于游戏，除了概念之外还要规划一次专门的 Image Gen 素材生成：透明的角色/状态精灵（sprite）或精灵表（sprite sheet）、地形/平台图块（tiles）、收集品、危险物、目标/检查点对象、道具，以及环境有纵深时的 2-3 个视差/背景层。HUD 文字、计分、控制、物理和碰撞保持代码原生。

对于概念，若属于以下情况就拒绝或迭代：对全界面需求却只给了 header、杂乱、平庸、重复、规格不足、不可读、过度装饰、带有未明确要求也不存在于参考里的 hero eyebrows/kickers/badges/pills 而偏离规范，或不切实际、无法忠实实现。

## 设计质量门槛（Design Quality Bar）

概念应当看起来像资深产品设计师做出的专业产品 mockup：

- 一个清晰的创意想法或视觉观点。
- 强有力的首屏，带有清晰的价值主张、产品信号和主要行动点。
- 整页的节奏感：section、状态、过渡和移动端视图感觉像作为一个系统被设计出来，没有重复的卡片堆叠或重复的 section 套路。
- 连贯的 section 到 section 的流动：用共享的间距、配色、字型节奏、媒体处理和微妙的过渡来连接各 section，而不是靠发明重大的新 UI 组件。
- 出色的排版：清晰的层级、尺度、字重、行高、标签处理，以及绝不回退到浏览器默认尺寸的控件/框架文字。
- 有意图的留白和密度；除非明确要求或存在于被接受的设计中，否则没有填充式卡片、hero eyebrow/kicker 标签、pills、badges、伪造指标或图标行。
- 默认更简：使用更少、更有力的视觉元素，而不是用插画、图标、装饰性小部件或复杂 UI 框架填满页面。
- 连贯的视觉系统：配色、间距、圆角、边框、阴影、渐变、图标风格、图像和组件几何。
- 有图标时，图标保真很重要。匹配被接受设计的图标隐喻、描边粗细、填充风格、拐角形状、尺寸、颜色、对齐和间距，而不是换上附近的通用图标。
- 颜色保真是强制的。匹配被接受设计的实际背景、表面、文字、边框、阴影和强调色；如果设计使用白色背景，就用白色，而不是米色（cream）、象牙白（ivory）、米黄（beige）、暖灰（warm gray）或任何软化过的 off-white 替代品。
- Hero 媒体的处理必须匹配被接受的设计。如果概念里的 hero 图没有颜色叠加或色调，实现就不能加。只用边缘淡出（edge fades）、遮罩（masks）或背景渐变来把图像边缘融进页面；不要用颜色叠加去冲刷图像。
- 为 logos、品牌标记、hero 图像、产品渲染、背景场景、插画、纹理、海报、头像、空状态，以及游戏精灵/图块/背景层，提供高质量的生成素材。产品/背景素材应完整渲染，带有一致的品牌，并在文字属于该素材时带上图内文字。
- 有目的的动效，用来厘清层级、揭示状态，或让产品感觉可触及。
- 当用户没有提供确切文案时，写具体、非通用的文案。

默认走干净、通透、有品位的 7/10 创意度：有辨识度到感觉是被设计过的，克制到能落地实现，且不重复。把「干净」理解为经过编辑且易读，而不是空洞或无菌。

## 视觉方向默认值（Visual Direction Defaults）

当用户没有给出更强的美术方向时，使用这些默认值。把它们适配到产品类型，而不是把每个应用都硬塞进营销站风格。

- 基线：大约 7/10 创意度、低到中密度、慷慨的间距、高实现清晰度、高排版纪律，以及当领域受益于真实视觉时以图像主导的时刻。
- 生成概念之前，选定一个连贯的视觉方向：一套主题范式、背景气质、排版气质、hero 或主屏幕架构、section/应用节奏、2-4 个标志性组件母题（motifs），以及 1-2 个动效线索。锁定这个组合，让设计感觉有意图，而不是通用模板。
- Hero 或首屏：保持一个明显的焦点、一句简短可读的标题或主要任务、克制的辅助文案、一个可见的主要行动点，以及在小尺寸笔记本上也够用的负空间。不要用统计数字、chips、badges、伪造控件或互相竞争的迷你面板挤满开场视图。
- Header 的简洁：默认用一个干净的品牌标记、必要的导航，以及一个主要行动点或控件。避免图标密集的导航、多余按钮、搜索栏、状态小部件、分段控件、装饰性插画或密集的产品框架出现在 header 里，除非用户明确要求或产品工作流需要。
- 视觉经济：优先用一到两个高质量的图像或插画时刻，而不是许多小的装饰性视觉。只在能厘清导航、控件或产品含义的地方使用图标。
- 容器纪律：避免嵌套卡片、把每个 section 都包在巨大的圆角容器里、默认的 bento/卡片网格，以及过度加框的仪表盘，除非概念或产品类型确实需要。优先采用开放布局、band、rail、列表、表格、画布，或单一有目的的加框动作。
- Section 节奏：长页面应在保持一套连贯品牌系统的同时，变化密度、图文比、对齐、尺度、留白和视觉节奏。不要在整页里重复同一个居中块或左文/右卡的套路。
- Section 连续性：当多个 section 概念需要合成一页时，用既有设计系统里的连接组织：gutters、bands、对齐、重复的排版、反复出现的媒体框、色彩节奏，以及小的过渡性间距变化。不要发明重大的新 carousels、accordions、定价卡片、仪表盘、表单、导航系统、功能网格或其他组件族，除非用户要求或被接受的概念里展示了它们。
- 媒体加框：生成的图像通常应放在清晰、便于实现的框里，带有稳定的宽高比、一致的裁切逻辑、圆角、阴影和间距。避免随意的图片尺寸或拼贴式的混乱，除非用户明确要求那个方向。
- UI 约束：小标签、utility pills、伪系统标记、伪造指标和装饰性的仪表盘术语，只有在能厘清产品时才允许。如果它们只是视觉填充，就在接受之前移除它们。

## 概念评审模式（Concept Review Mode）

只在用户要求先生成概念、评审多个选项或等待批准时使用。

- 生成并展示概念。
- 迭代直到用户批准。
- 在用户仍在评审时不要实现。
- 一旦被批准，把该概念当作当前生效的规范，并遵循下面的保真工作流。

## 构建前（Before Coding）

在写代码之前，把被接受的概念转成一套设计系统和一份实现清单：

- 确切的可见文案、导航项、CTA 标签、section 标题、佐证要点（proof points）、数据标签，以及重要的 UI 文字。
- 逐 section/状态的图像清单：来源概念截图、原生宽高比、视觉优先级、可读文字、排版关系、间距、按钮/控件样式、组件/容器规则、主导颜色，以及任何需要重新提取截图才能解决的未决细节。
- 允许的首屏文案清单：从被接受的概念或用户提供的文案中允许出现的每一条可见 hero、导航、eyebrow/kicker/pretitle、badge/pill、CTA、标签和佐证字符串。
- 首屏构图、section 顺序、下游状态、响应式延续，以及下一 section 的预览。
- Section 连续性计划：相邻 section 如何用被接受的设计系统连接，以及哪些主要组件族是被允许的。把未展示的主要组件视为禁止，除非用户要求，或某个必需的工作流没有它就无法运作。
- 品牌标记、图像角色、产品 mockups、仪表盘、表格、图表、地图、媒体 rail、表单、HUD，或其他视觉产物。
- Hero/媒体处理清单：每张图是无叠加、颜色叠加、渐变叠加、边缘淡出、遮罩、透明背景，还是匹配的背景色。在写代码之前明确记录这一点。
- 独立素材需求：如果概念包含 logo、品牌标记、产品标签、包装、海报、招牌、产品渲染或带品牌的背景对象，就在实现前用 Image Gen 编辑创建匹配的独立素材，让品牌保持连贯。
- 游戏素材需求：如果概念是游戏，就在实现前用 Image Gen 创建匹配的生产美术素材。按需包含透明的精灵/状态素材、图块/平台、收集品、危险物、目标对象、道具，以及视差/背景层；碰撞盒和游戏状态用代码，不要拿代码替代可见美术。
- 从图中采样或近似得到的设计 tokens：背景、表面、文字、弱化文字（muted text）、边框、阴影、强调色、语义色、圆角、层次（elevation）、间距刻度，以及动效时序。
- 颜色锁定：明确判定概念背景是纯白、off-white、米色、灰、深色还是带色调，然后实现那个确切选择。不要把配色调暖、调冷、调暗，或以其他「有品位」的方式重新诠释。
- 排版系统：字体族/回退、字号刻度、字重、行高、字距（tracking）、标签处理、标题/正文/说明文字样式、控件文字样式，以及响应式字型行为。
- 图标清单：每一个可见图标、字形、chevron、类 logo 标记、工具栏符号、状态符号和空状态符号；记录含义、来源字体族、outline 还是 filled 风格、描边宽度、尺寸、颜色、容器、对齐、间距，以及 selected/hover/disabled 的处理。
- 组件族与变体：按钮、导航、行（rows）、面板、媒体框、产品 mockups、仅在出现处的卡片、表格、表单、chips、图标、空状态、响应式变体、hover/focus/selected 状态。
- 复杂应用 UI 的组件架构计划：应用外壳（app shell）、导航、主要功能区、可复用 UI 原语、数据/状态辅助、图表/表格/表单模块、素材模块，以及响应式布局边界。一个优秀的前端实现应有清晰的组件归属，而不是一个巨大的 `App` 组件或一次性拷贝的标记。
- 容器模型：卡片、面板、rail、band、列表、表格、画布、drawer、侧栏、模态框，或全出血（full-bleed）section。
- 核心工作流：必须响应的控件、selected 状态、筛选器、tabs、编辑、创建流程、成功状态、播放、游戏控制，或生成结果的演示。

如果概念遗漏了必需的下游 section、状态、移动端视图，或复杂区域的可读细节，当视觉一致性或提取存在不确定性时，就生成匹配的 section/状态/细节概念。否则就在完全相同的视觉系统里延展。

## 实现（Implementation）

- 先构建真正可用的界面，而不是围绕一个未来应用的营销外壳。
- 遵循仓库的框架、路由、组件、样式、状态、无障碍和素材约定。
- 当在没有既有框架约束的情况下创建全新的复杂应用 UI 时，默认使用 React + Vite。像资深前端工程师那样组织它：小而聚焦的组件、清晰的应用外壳、可复用的原语用于重复控件、针对仪表盘/表格/图表/表单的功能专用模块、分离的示例数据和状态辅助，以及共享的 tokens/样式。让 `App` 保持为组合胶水，而不是一个庞大的单屏实现。
- 通过从图中提取的设计系统来实现。相似元素必须使用相同的组件或共享样式原语；差异应是显式的变体，而不是一次性拷贝的 CSS。
- 精确实现被接受的概念。保留文案、层级、section 顺序、密度、颜色、排版、间距、圆角、边框、阴影、素材加框和交互模型。
- 对于多 section 页面，按匹配被接受 section 概念的切片来实现。从首屏开始，把它的浏览器截图和 section 概念对比，修复可见偏差，然后逐个 section 继续。不要把所有视觉对比都推迟到整页编码完成之后，也不要仅仅因为一张宽泛的总览图更容易跟随，就合并或简化 section 专属的设计决策。
- 把各 section 连成一个连贯的页面，且不添加未经批准的主要 UI 组件。用间距、背景 band、对齐、排版节奏、重复母题和媒体加框来弥合空隙。不要发明新的 carousels、accordions、定价块、仪表盘、表单、tab 系统、功能卡片网格或其他大型组件来让页面感觉完整，除非它们出现在被接受的概念里、由用户要求，或被记录为一个具体的功能必要性。
- 概念被接受后，不要添加新的可见首屏文案、hero eyebrows/kickers、解释性标签、副标题或分类文字，除非它出现在被接受的概念里、来自用户，或被记录为一处有意的偏离。如果语义化 HTML、SEO 或无障碍要求改变某个 H1 或标题级别，先改元素语义；不要发明补偿性的可见文案。
- 不要添加不在被接受设计中的装饰性 hero eyebrow 标签、pills、badges、渐变、光晕（glows）或叠加层。不要替换成某个渐变处理，除非它匹配概念的配色、方向、强度和位置。如果被接受的 hero 图没有颜色叠加，就不要在它上面加半透明色调、冲刷或有色图层。如果图像需要帮助才能融进不匹配的页面背景，就用匹配的素材、透明抠图、边缘淡出、遮罩，或围绕图像的背景渐变，而不是在图像上叠一层颜色。不要把白色背景换成米色/off-white，或以其他方式移动被接受的色温。
- 有意地为控件定义排版。不要为按钮、tabs、inputs、工具栏、侧栏、检查器面板、图层行、状态栏、命令面板（command palettes）或导出/分享控件依赖浏览器默认值或继承的 `16px` 尺寸。
- 保留容器模型。不要在规范使用开放留白、band、rail、列表、表格、画布或全出血构图的地方添加卡片、带边框的面板、浮动容器、瓦片或卡片网格。
- 让真正可交互的应用 UI 文字、导航、按钮、表单、表格、控件和标签保持代码原生。这不适用于属于产品图像、海报、包装、招牌、背景场景、hero 照片或其他栅格素材内部的文字和品牌。不要把静态截图当 UI 出货。
- 对于中心的非图标素材使用 Image Gen。把产品图像和背景素材完整渲染出所需的文字、logos、标记、标签、包装、招牌和品牌。当素材必须层叠进 UI 时，请求透明背景或干净的抠图。引用确切的素材文字，并在文字重要时要求逐字渲染。
- 如果被接受的设计包含带品牌的产品图像，就用 Image Gen 编辑从概念或匹配的素材生成中创建 logo/产品/包装/招牌素材的独立版本。当这些素材需要层叠进 UI 时，包含透明背景变体。不要用通用 CSS、不匹配的字体或近似标签来重建带品牌的栅格素材。
- 对于游戏，用 Image Gen 生成可见的生产美术：角色/状态精灵或精灵表、地形/平台图块、收集品、危险物、目标/检查点、前景道具，以及视差/背景层。不要因为碰撞、缩放或动画更简单就退回到画布绘制的形状。让 HUD 文字、分数、控制、碰撞盒、物理和游戏状态保持代码原生，并把碰撞几何调到与渲染出的素材吻合。任何代码绘制或矢量的游戏美术都必须列为一处有意的偏离或具体的阻塞问题。
- 不要用粗糙的 CSS 绘图、通用渐变、占位 SVG 或类库存图的裁切来替换概念素材。图像必须自然地坐落在构图里：背景色、光照、边缘、裁切、阴影和透明度应与周围设计融合。对于忠实的图标和方向字形，SVG 是可以的。
- 对箭头、chevrons、carets、展开指示符、分页箭头和轮播箭头，使用 SVG/图标组件；不要使用纯文本字形，除非概念有意为之。
- 像对待其他视觉元素一样忠实地实现图标。仅当仓库既有的图标集或 lucide 匹配被接受设计的风格时才优先使用它们；否则创建一个匹配概念的小型自定义 SVG/图标变体。自定义 SVG 图标必须是生产级矢量素材：清晰的 `viewBox`、干净的几何、一致的描边宽度、对齐的连接/端点、平衡的负空间、视觉居中、可缩放的路径、没有锯齿或占位感的形状，且仅在匹配设计系统时使用 `currentColor` 或显式填充。不要把 filled 图标换成 outline 图标、把圆润图标换成尖锐图标、把粗描边换成细描边，或把具体隐喻换成通用符号。让图标颜色、视觉尺寸、基线对齐、内边距和交互状态与提取出的图标清单保持一致。
- 让应用界面具有体验性：本地 state、有意义的 selected 状态、可用的筛选器/tabs/表单、可编辑或可创建的项、成功状态、播放控制、游戏控制，或恰当处的模拟生成输出。
- 只在真正合适时才在 hero 内部使用可交互 UI：SaaS/软件产品预览、产品演示，或有目的的交互动画。不要把假的交互框架硬塞进带品牌的、编辑向的、产品、场地、食物、消费或背景主导的 hero。忠实的实现和一致的品牌比添加交互性更重要。
- 只在支持设计的地方添加动效。尊重无障碍和 `prefers-reduced-motion`。
- 让实现保持面向生产：语义化标记、稳定的响应式尺寸、没有脆弱的硬编码 hack，并在仓库支持时做类型/lint/测试检查。

## 验证（Verification）

运行应用并验证可见的产品，而不只是构建。

1. 优先使用 Browser/IAB。加载应用，检查首屏，滚动，并点击走完核心工作流。
2. 检查桌面、当前浏览器视口，以及一个移动尺寸的视口。
3. 捕获或定位被接受的概念和最新的实现截图。在最终交付前的同一次 QA 中，对两者都使用 `view_image`；不要跳过这一步，也不要用一眼浏览器来替代它。
4. 在可行时，以被接受概念的原生尺寸捕获实现。若不可行，记录阻塞原因，并同时验证当前浏览器视口。
5. 在最终确定前写一份保真台账（fidelity ledger）：偏差、概念证据、渲染证据，以及已做的修复或未修复的理由。对于多 section 或多状态的规范，要包含来自相关 section/状态概念截图的证据，而不只是总览图。检查至少五个具体的对比点，覆盖文案、布局、排版、配色/渐变、素材处理、间距/容器模型、响应式行为或动效。
6. 并排对比：文案、导航、CTA 标签、section 顺序、首屏平衡、下一 section 的可见性、配色、渐变处理、字体性格、字号刻度、间距、边框、圆角、容器模型、素材/背景融合、动效，以及模拟的交互。
7. 对照允许的文案清单做一次首屏文案 diff。被添加、移除、改名或重排的可见文案必须被修复或列为有意的偏离；未经批准的添加视为保真不合格。
8. 在所有地方审计排版，不只是 hero 或主画布。检查标题、正文、说明文字、标签、工具栏控件、侧栏行、tabs、inputs、检查器字段、状态栏、命令面板、导出/分享按钮、表格单元、图表标签，以及移动端的换行。当截图暗示有偏差时，使用计算后的 CSS 尺寸/字重/行高。
9. 在图标出现的任何地方审计图标：导航、按钮、卡片、工具栏控件、侧栏、表格、状态指示、空状态、分页、轮播和移动端控件。对照被接受的概念检查隐喻、描边/填充风格、尺寸、颜色、对齐、视觉重量、间距和状态变化。
10. 对于画布/编辑器类应用，把应用框架与画布/文档文字分开审计。默认的缩放和平移是规范的一部分；持久化的本地 state 不得在验证期间隐藏 seed、缩放或排版的修复。
11. 明确地问：这是否达到设计公司验收标准的忠实实现，一家优秀、技艺高超的设计公司是否会认可对被接受设计的这一具体实现？如果有任何东西会收到设计评审意见，就写一份具体的修复清单并继续编辑。不要在还有可修复的视觉问题时给出最终答案。
12. 验证生成的素材能加载、加框正确，且不遮挡文字或控件。
13. 验证核心工作流会更新真实的本地 UI state。不要出货惰性控件、假的媒体进度、隐藏的必需媒体或占位交互。

功能性 QA 不等于保真 QA。通过构建检查、点击控件或验证本地 state，不能替代概念到截图的对比、原生尺寸检查和书面的偏差台账。

硬停止项（Hard stops）：主要内容被裁切、意外换行、原型感的布局、粗糙的种子数据、占位方块、类库存的通用素材、未完成的卡片、代码绘制的游戏占位物替代了概念美术、发明的可见文案、发明的 hero eyebrows/kickers/pills/badges、不匹配的颜色或渐变、白色背景被改成米色/off-white、未经批准的 hero 图颜色叠加或色调、缺失或被通用替代的图标、不匹配的图标风格或描边粗细、与背景不融合的图像、陈旧的调试产物、不可读的文字、字号刻度偏差、浏览器默认的控件排版、移动端溢出、不专业的响应式坍塌，或任何偏离被接受规范的可见漂移。

## 界面门槛（Surface Gates）

- Landing/公司站：保留首屏、hero 角色、品牌/导航/CTA 标签、section 顺序、下一 section 预览，以及标志性图像。
- 产品/SaaS 页面：保留产品 mockups、工作流图示、功能条、佐证元素和品牌处理。
- 仪表盘/工具：保留密度、侧栏、header、表格、tabs、时间线、图表、地图、行数，以及 selected/详情行为。不要把表格驱动的概念变成卡片网格。
- 画布/编辑器工具：保留默认缩放/平移、画布/文档文字比例、框架密度、工具栏、侧栏、检查器控件、图层行、状态栏、命令界面，以及自动保存/种子状态行为。
- 时间线/规划工具：保留网格/时间轴结构、行跨度（row spans）、事件密度、状态 rail，以及指挥中心（command-center）式的贴合。
- 类克隆界面：在加抛光之前保留可辨识的骨架。不要添加营销 hero 或破坏产品类型的自定义导航。
- 游戏：用 Image Gen 素材保留美术方向——精灵、图块/平台、收集品、危险物、目标/检查点、道具，以及背景/视差层。验证素材能加载、缩放、动画或切换状态正确、与碰撞几何对齐，并支持移动、动作/跳跃/拖拽行为、计分、危险物和重开。
- 媒体界面：验证真实媒体能加载、时长、播放/暂停、快进/进度，以及可见的帧变化。
- 表单/预订/购买/餐厅流程：验证主交易路径和确认状态。
