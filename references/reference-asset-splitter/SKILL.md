# Reference Asset Splitter

## 目的（Purpose）

在任何生成或编码开始之前，把视觉参考转化为一份经过确认的素材拆分方案。防止两种失败模式：把一个融合在一起的静态场景过度拆分成许多碎片，以及把动态内容过度合并成一张死图。

这个 skill 不生成图片、不裁剪截图、也不实现站点。它产出的是一份面向用户的清单，必须先获得批准，才能开始素材生成。

## 唯一原则（The One Principle）

**由交互性和可编辑性决定边界——而不是视觉复杂度。**

复杂 ≠ 素材。卡片 ≠ 代码。一个视觉上很复杂的组合，只要其中没有任何东西需要独立行为，就可以是一张图；一个看起来很简单的卡片，只要它的文字需要编辑，就必须留作代码。永远不要按某样东西看起来有多难来分类；要按它在站点建成之后将如何被使用来分类。

默认采用**在保真的前提下最粗的素材**，只有在命中强制拆分触发条件时，才进一步拆得更细。

## 决策流程（Decision Flow）

自上而下、按此顺序进行。不要一上来就枚举可见对象——枚举是最后一步，不是第一步。

### 1. 读区域，而不是读对象（Read the region, not the objects）

对于页面的每个区块，在查看单个组件之前，先判断它的用途类别：

- **static-decor**：一个融合在一起的装饰场景或卡片组合，没有任何独立行为。对象彼此重叠、共享阴影/纹理，只为营造氛围而存在。
- **content-card**：以文字/排版为主的区块，其文字必须保持可编辑、可选中或事实正确（个人简介卡、技能列表、时间线、仓库卡、指标数据）。
- **media-slot**：承载真实视觉内容的独立可重复槽位——照片、产品/UI mock 截图、视频封面。每个槽位都有自己独立的含义，可以被独立替换。
- **identity**：真实头像、logo、个人照片、事实性指标——永远不要作为最终成品生成；标记为素材来源或稍后替换。

### 2. 选择粒度（Choose granularity）

- static-decor → **一张粗颗粒素材**覆盖整个融合组合（背景形状 + 纹理 + 装饰卡片一起）。
- content-card → **代码原生的外壳与文字**；只把一处难以用代码实现的子材质（印刷图、线描、插画、纹章、纹理）提取成它自己的小素材。
- media-slot → **每个有意义的槽位一张素材**。绝不把独立的照片/mock 合并成一张拼合图；也绝不把一张 mock 内部拆成几块。
- identity → 素材来源或稍后替换。

强制拆分触发条件（唯一能让你拆得比默认更细的理由）：

1. 某个组件需要独立交互：hover、点击、展开、滚动、动画。
2. 某个组件之后需要被独立替换或更新。
3. 某个组件包含可编辑或事实性文字（那么这些文字回归代码）。
4. 某个组件必须与相邻组件分开、独立地响应式重排。

强制合并守则（唯一能让你拆得更粗的理由）：多个组件在视觉上融合——彼此重叠、互相投射阴影、共享纸张/纹理的连续性——并且以上任何触发条件都不适用。

### 3. 两个对称的陷阱（Two symmetric traps）

- **碎片陷阱（Fragment trap，过度拆分）**：把一个融合的静态组合拆成逐张卡片素材，只因为每张卡片单独看都能被辨认出来。如果这些卡片是装饰、而且没有任何东西需要动，那它就是一张图。
- **面板陷阱（Panel trap，过度合并）**：把一个带底色的容器面板压成一张图，只因为它「看起来很复杂」。一个装着文字卡片的面板是一个布局，不是一张图片：面板主体、渐变和圆点纹理是代码；文字卡片是代码；只有其中真正难以用代码实现的材质（一枚纹章印刷图、一幅线描、一张照片）才变成一张小素材。

### 4. 交叉核对枚举（安全网，最后做）（Cross-check enumeration）

现在扫描每一个可见的非文字单元，核实每一个都恰好被某一个决策覆盖：包含在某张粗颗粒素材里、成为自己的素材、代码原生、或稍后替换。这一步能抓住网格里漏掉的照片和意外被烘焙进去的文字。它绝不能推翻第 1–2 步的分组。

### 5. 陈述假设；歧义时分叉（State assumptions; fork when ambiguous）

当仅凭参考无法判断某个区域是静态装饰还是可交互内容时，输出一个**推荐默认方案外加一个备选方案**，并说明各自基于哪个假设。示例：「默认按静态装饰处理，整块一张背景图；如果这些卡片需要 hover/点击/替换，则改为拆出 N 个卡片素材。」

绝不对有歧义的区域悄悄选定一种解读。

hero 一侧的装饰性拼贴组合**永远**要做这个分叉——把「整个组合作为一张背景」和逐卡片拆分并列呈现，即便这些卡片里含有看起来像事实的数字、统计或产品名。在个人 IP 的 hero 里，这些数字通常是装饰性氛围，而不是用户维护的数据；不要让它们逼出一次悄悄的拆分。推荐一个选项，但绝不省略另一个。

## 分类标签（Classification Labels）

- `background asset`：整区块或大图层图片，承载纹理、氛围、照片真实感，或一个融为一体的场景。
- `foreground asset`：一个离散的图像对象——肖像、产品截图、卡片照片、拼贴、设备屏幕、难以用代码实现的印刷卡片。
- `source asset`：一张已存在的真实图片，用来复制或链接，而非生成。
- `code-native`：文字、导航、按钮、文字卡片、边框、简单图标、分隔线、箭头、时间线、标签页、阴影、渐变、圆点网格、布局装饰件。
- `replace-later`：真实的身份信息、logo、照片、指标或私人事实，目前是占位/假数据，绝不能当作最终真相处理。

不要把重要文字、导航、按钮或动态内容烘焙进图片素材。素材内部类似文字的装饰性标记只有在非关键时才可接受，且方案必须说明这一点。

除非用户明确要求，否则不要把参考截图裁剪后作为最终构建素材。

## 强制流程（Mandatory Flow）

第一步 / 素材拆分清单：检视参考截图并产出拆分方案。除非已获批准，否则在此停下等待用户确认。

第二步 / 按清单生成素材：批准后，精确地生成或收集已批准的素材。如果出现新需求，回到清单。

第三步 / 素材审核：展示生成/收集到的素材，标注假数据或稍后替换的项，等待批准。

第四步 / 最后复刻：只有在素材获批之后，才继续进入图生代码的实现。

使用这些自然的步骤标签。绝不在面向用户的进度里使用 `1/3`、`2/3` 这类计数器。

## 必需输出（Required Output）

用用户的语言作答（中文消息就用中文），当用户给出多个页面时逐页作答——绝不为「每一页」这类请求给出一张合并的大表。

```text
第一步 / 素材拆分清单

【第 N 页 · 页面名】

交互假设：<这一页哪些区域按静态装饰处理、哪些按可编辑内容处理，一句话>

总判断：需要 N 个图片素材。

| ID | 类型 | 状态 | 内容 | 边界 | 代码叠加 | 判断理由 |
| --- | --- | --- | --- | --- | --- | --- |
| hero-visual | background asset | generate | ... | ... | ... | ... |

Code-native：
- ...

Replace-later / 风险：
- ...

（如有歧义）备选方案：若 <交互假设变化>，则 <改动>。

我会先停在这里。你确认后，我再按这份清单生成或整理素材。
```

如果用户问「对不对」或提供了标注截图，就明确地校验他们提出的边界，并给出理由加以修正——用户的红框标注定义了他所期望的分组。
