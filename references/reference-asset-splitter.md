# Reference Asset Splitter

## 任务

在生成素材或编写页面代码前，把每张已确认参考图转成一份已确认、可实施的素材方案。既要保留参考图质量，也要避免把融合画面过度切碎，或把实时内容烘成死像素。

不要生成图片、裁切参考图或实现页面。只产出拆分裁决并停下等待确认。已确认参考图始终是 QA 目标，不是构建素材。

完成拆分前读取同级 `split-patterns.md`，匹配最接近的先例。

## 必需输入与默认值

有条件时检查：

- 每个页面或 section 的已确认参考图
- 已确认内容与风格 brief
- 交互、替换、动画和移动端 reflow 要求
- 已知头像、logo、截图、照片和官方源文件
- 用户的红框、分组名称等标注

把用户标注视为明确分组意图；如果与真实性、语义内容或必要行为冲突，要解释原因。

没有说明交互时，直接使用以下默认值，不自动追问：

- 融合装饰区域按静态处理，保持粗颗粒
- 页面文案、导航、链接、按钮和事实内容保留为代码
- 有独立含义的照片、头像、视频封面或产品截图保持可独立替换
- 不因行为未说明就额外提供一套拆分；Hero 没有特殊例外

只有存在真实歧义时才提问或提供备选：同一区域既可能是静态画面，也可能是需独立维护的内容；brief 无法裁决；选择会显著改变素材数量或构建行为。

## 固定冲突优先级

严格按下列顺序处理冲突，低优先级不得覆盖高优先级：

1. **真实性（Authenticity）**：保护真人身份、官方品牌、真实作品、私密事实和承载主张的截图。使用已验证来源、`replace-later` 或 `blocked`；绝不伪造成最终素材。
2. **语义归属（Semantic ownership）**：有意义文案、导航、control、链接、label、已确认指标和事实数据保留为代码。未批准的文字式装饰改成不可读抽象标记，不能发明主张。
3. **交互与 reflow**：需要独立 click、hover、animate、update、replace、scroll 或响应式移动的内容必须拆开。
4. **渲染媒介**：执行下述 code whitelist 与 raster signals；混合区域使用 `hybrid`。
5. **融合（Fusion）**：只合并剩余且属于同一不可分视觉组件的 raster 部分。
6. **素材预算**：减少不必要装饰素材，但不能合并有语义或交互独立性的媒体。
7. **品味（Taste）**：满足全部实施约束后，再保留已确认构图。

如果高优先级要求与参考图保真无法通过 hybrid 边界共存，返回 `redesign-required`，不要选择好看但不可实施的拆分。

## 三轴裁决

对每个有意义区域都执行三轴分析，不要先数可见对象。

### A 轴——边界颗粒度

先识别区域职责：

- **live-content**：可编辑文案、UI、数据、导航或 control
- **identity/source**：头像、logo、官方标记、真实项目图或事实截图
- **independent-media**：可以在不改变邻居的情况下替换的照片、mock、封面或视觉槽
- **static-decor**：没有独立含义/行为的氛围、装饰或融合场景

只有命中至少一个强制拆分条件时，才建立独立边界：

1. 独立交互或动画
2. 独立替换、来源或未来更新
3. 必须保持实时的语义/事实内容
4. 独立响应式 reflow 或裁切
5. 与周围生成画面不同的真实性归属

其他静态 raster 装饰保持能保留已确认构图的最粗边界。卡片、纸张、窗口、物件或道具仅仅“可识别”，不代表必须单独成文件。

### B 轴——渲染媒介

只有所有可见部分都在以下白名单中时，才能标为 `code-native`：

- 实时文字和标准网页排版
- 布局容器、间距、grid、column 和响应式定位
- 纯色和普通透明叠层
- 简单线性/径向 gradient
- border、radius、divider、标准 shadow 和简单 mask
- 基础 line、dot、ring、arrow、pill 和重复 pattern
- 标准 icon，以及几何可精确复现的简单 SVG diagram
- tab、button、card、label、form control 等普通 UI chrome

以下 raster signals 要求视觉核心使用图片素材：

- 摄影或写实场景
- 绘画、手绘、编辑式或材质不规则插画
- 纸张颗粒、布料、墨迹、印刷磨损、自然纹理等复杂材质
- 复杂光线、反射、体积效果、真实景深或 3D rendering
- 设备/产品 render、密集 UI screenshot、承载主张的视觉证据
- 分层拼贴、真实桌面场景、复杂地图或生成艺术
- 需要描摹或只能用近似代码替代的不规则形状/pattern

混合两类时使用 `hybrid`：外壳、实时文字、control 和交互留在代码里；只有视觉核心使用一个或多个 raster 素材。不能因为一个局部需要 raster 就把整个区域做成图片。

### C 轴——图片层级

每个 raster 素材必须且只能选择一个层级：

- **`background`**：位于代码内容后面的 full-section 或 edge-to-edge 环境表面；必须能承受响应式裁切，不能包含实时 UI 或有意义文案。
- **`integrated-scene`**：section 内一张有边界、不透明的融合构图，例如 Hero 侧边世界、地图平面、桌面 tableau 或编辑式拼贴；内部整体移动/裁切。
- **`composite-foreground`**：背景上方的独立或透明融合簇，例如互相遮挡的物件 vignette；内部共享遮挡、光线或阴影，没有独立行为。
- **`independent-media`**：有独立含义、来源、替换或响应式需求的媒体槽，例如头像、项目截图、照片、视频封面、logo、产品 mock。

`source`、`generate`、`replace-later` 描述来源/状态，不是图片层级。

复用范围另记为 **`global`** 或 **`section`**。纸面、grain field、scan strip 等全站 fingerprint 材质可生成一次后复用。shared material 不是第五个层级，不能把 section 专属视觉改标为 global 来降低表面素材数量。

## 构建融合图（Fusion Graph）

真实性、语义、交互和媒介裁决完成后，才能构建 fusion graph：

1. 从 raster 分组中移除 code-native 内容。
2. 把 identity/source 和强制拆分项锁为独立节点。
3. 为剩余可 raster 化部分建立节点。
4. 两节点存在以下强融合关系时连边：跨对象遮挡、投影/反射、连续材质/纹理、同一透视平面、不可分光线或依赖无缝边缘的绘画。
5. 不能仅因为相邻、对齐、同处一个 panel 或配色相同就连边。
6. 每个连通分量默认成为一个 raster 素材，再分配 C 轴层级。

不要为了缩小文件而切断强融合边；也不要让融合边跨越更高优先级的独立媒体/交互边界。两者都必须保留时，标记 `redesign-required`。

## 素材预算

允许 0 个素材。每个 section 最好至多一个装饰性 `background`、`integrated-scene` 或 `composite-foreground`，再加语义、真实性、交互或替换真正要求的 `independent-media`。

一个 section 超过一个融合装饰素材或三个独立媒体时重新审查。这是过度拆分警告，不是硬配额；不能为了达标而合并真实或独立有意义的媒体。

## Redesign Required

已确认像素无法在不破坏高优先级规则的情况下拆解时，返回 `redesign-required`：

- 需要独立交互的部分依赖跨边界阴影、反射或遮挡
- 真人头像、logo 或事实截图不可逆地融合进生成场景
- 有意义文案被画在透视表面/纹理上，无法既保持实时又保留构图
- 移动端要求融合 raster 场景内部元素独立重排
- 精确复刻需要截图切片、脆弱像素接缝或代码近似核心视觉

写明冲突区域、冲突要求和保留视觉想法的最小参考图重设计。在修订参考图确认前，停止该区域的生成或实现。

## 文字与来源安全

- 不把导航、button、link、重要 label、已确认指标或有意义文案烘进素材。
- 生成画面内只允许无语义的文字式细节；要求抽象不可读 bar/mark。
- 真人头像、logo、官方标记和真实项目截图保持已验证 `source` 或 `replace-later`；替身会误导时使用 `blocked`。
- 用户未明确要求低保真复用时，不得裁切、切片或截图已确认参考图作为最终素材。

## 状态词汇

只使用以下状态：

- `code-native`：按白名单用代码实现
- `generate`：根据参考区域生成不承载事实主张的 raster 素材
- `source`：使用已验证真实/官方素材
- `replace-later`：使用明确命名的临时替身，等待真实来源
- `blocked`：必须获得真实来源，暂不继续
- `redesign-required`：生成或实现前修改参考图

## 强制工作流

- `第一步 / 素材拆分清单`：检查每张参考图，执行三轴裁决，停下等待确认。
- `第二步 / 按清单生成素材`：确认后只生成/收集已批准素材；出现新需求时返回清单。
- `第三步 / 素材审核`：对照参考图检查准备好的素材，并等待确认。
- `第四步 / 最后复刻`：素材确认后才能进入 image-to-code。

使用自然语言阶段名，不要在用户可见进度中使用 `1/3` 式计数。

## 必需输出

内部完成完整三轴推理，对用户只给简洁的逐页清单。使用用户语言；多页时每页一个 section，不要把所有页面合并进一张表。

中文请求使用：

```text
第一步 / 素材拆分清单

【第 N 页 · 页面名】

交互假设：<静态区域、可编辑区域、独立媒体与移动端行为，一句话>
总判断：需要 N 个图片素材；其中背景/融合场景 N 个，独立媒体 N 个。<若为 0，说明全部可由代码高保真实现。>

| ID | 图片层级 | 复用范围 | 状态 | 内容与边界 | 代码覆盖 | 裁决理由 |
| --- | --- | --- | --- | --- | --- | --- |
| hero-world | integrated-scene | section | generate | ... | 标题、CTA 保持代码 | A: 静态融合；B: 光影/材质需图片；C: 区域内场景 |

Code-native：
- ...

真实来源 / Replace-later / 风险：
- ...

（仅在触发时）Redesign-required：<冲突 + 最小重出方案>
（仅有真实歧义时）备选方案：若 <未决行为>，则 <边界变化>；推荐 <默认>，因为 <理由>。

```

用户提供标注截图或询问拆分是否正确时，逐项对照固定优先级验证边界，并给出具体纠正理由。

## 最终验证

展示清单前检查：

1. 按 authenticity → semantics → interaction → medium → fusion → budget → taste 执行优先级。
2. 为每个有意义区域分配 A、B、C 三轴结果。
3. 每个可见非文字单元只归属一次：融合素材、独立素材、code-native、source/replace-later、blocked 或 redesign-required。
4. 除非高优先级边界迫使重设计，保持 fusion graph 每个连通分量完整。
5. 所有实时和事实内容都不得进入生成像素。
6. 真人身份与承载主张的媒体使用诚实来源状态。
7. 无交互说明时使用静态粗颗粒默认值；只有真实歧义才给备选。
8. 重新检查素材预算，但预算不得覆盖语义；每个 unique global shared material 全站只计一次，并列出复用位置。
9. 分页展示并停在确认点。
